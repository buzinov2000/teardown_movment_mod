## 0. Project Overview

**Что это:** мод для игры Teardown.

**Стек:**
- Lua 5.1 (именно 5.1 — никакого `goto`, никаких bitwise операторов из стандартной библиотеки, никакого `//`, никаких `continue`)
- Teardown API — лежит в `Docs/api/`. Точка входа: `Docs/api/00-index.md` (карта всех 607 функций по категориям + алфавитный индекс со ссылками на якоря)
- XML для уровней, префабов и описаний объектов
- VOX-файлы для моделей (MagicaVoxel)

**Точки входа в `main.lua`:**
- `init()` — один раз при загрузке мода
- `tick(dt)` — каждый физический тик (важно для физики/инпута)
- `update(dt)` — каждый кадр (важно для логики, не привязанной к физике)
- `draw()` — UI и оверлеи через `UiPush/UiPop`

Логирование — `DebugPrint(...)`, не `print()`.

## 1. Workflow по умолчанию

При любом запросе, требующем кода с API Teardown:

1. Открой `Docs/api/00-index.md`, найди нужные функции (либо в категориях, либо в алфавитном списке)
2. Перейди в соответствующий категорийный файл (`Docs/api/NN-category.md`) и сверь сигнатуру, типы параметров, дефолтные значения
3. Только после этого пиши код
4. После написания — кратко перечисли, какие API-функции были использованы, чтобы я мог быстро верифицировать

**Это правило не опционально.** API маленькое и специфичное — галлюцинации тут главная причина нерабочего кода. Если функция не нашлась в индексе — значит её нет в API, останавливайся и спрашивай.

## 2. Think Before Coding

**Don't assume. Don't hide confusion. Surface tradeoffs.**

Перед тем как писать код:
- Явно проговори допущения. Если не уверен — спроси.
- Если возможны несколько интерпретаций задачи — покажи их, не выбирай молча.
- Если есть путь проще — скажи. Пушбэк уместен, если он обоснован.
- Если что-то непонятно — стоп. Назови что именно. Спроси.

**Teardown-специфика:**
- **Никогда не выдумывай функции Teardown API.** Сверка с `Docs/api/` — обязательна (см. секцию 1).
- Координаты, кватернионы, Transform — это области, где легко напутать. Если работаешь с пространственной математикой, проговори какие системы координат и оси используются.

## 3. Simplicity First

**Минимум кода, который решает задачу. Ничего спекулятивного.**

- Никаких фич сверх запрошенного.
- Никаких абстракций для одноразового кода.
- Никакой "гибкости" и "конфигурируемости", которую не просили.
- Никакого error handling для невозможных сценариев.
- Если получилось 200 строк, а можно 50 — переписывай.

Проверочный вопрос: «Сеньор сказал бы, что это переусложнено?» Если да — упрощай.

**Teardown-специфика:**
- Прототип ≠ продакшн. Хардкодь значения, пока механика не доказала жизнеспособность.
- Не строй системы конфигов и менеджеров до того, как появилась вторая сущность, которой они нужны.
- `registry` Teardown'а уже даёт persistent state — не изобретай поверх него свою систему сохранений до необходимости.

## 4. Surgical Changes

**Трогай только то, что должен. Убирай только свой мусор.**

При редактировании существующего кода:
- Не «улучшай» соседний код, комментарии, форматирование.
- Не рефактори то, что не сломано.
- Соблюдай существующий стиль, даже если ты бы сделал иначе.
- Заметил несвязанный мёртвый код — скажи, не удаляй.

Когда твои изменения создают сирот:
- Убери импорты/переменные/функции, которые твои изменения сделали неиспользуемыми.
- Не удаляй заранее существующий мёртвый код без просьбы.

**Тест:** каждая изменённая строка должна напрямую трассироваться к запросу пользователя.

## 5. Goal-Driven Execution

**Определи критерии успеха. Крути цикл до подтверждения.**

Превращай задачи в проверяемые цели:
- «Добавь телепорт» → «Нажатие T перемещает игрока в точку, куда смотрит камера, проверка — войти в игру и нажать»
- «Почини баг» → «Воспроизведи баг по шагам, потом сделай чтобы шаги давали корректный результат»
- «Сделай рефактор X» → «До и после — мод запускается и эта механика работает идентично»

Для многошаговых задач — короткий план:

```
1. [Шаг] → проверка: [как убедиться]
2. [Шаг] → проверка: [как убедиться]
3. [Шаг] → проверка: [как убедиться]
```

**Teardown-специфика:**
- Главный цикл верификации: написал → Play в редакторе → посмотрел консоль и поведение → правка. Не пиши по 300 строк без захода в игру.
- Если изменение можно проверить за 10 секунд в редакторе — попроси меня это сделать перед тем как двигаться дальше.
- `DebugPrint`, `DebugCross`, `DebugLine` — твои друзья. Используй их щедро на этапе сборки фичи и подчищай в конце.

## 6. Teardown-Specific Pitfalls

Грабли, которые я (Claude) должен помнить:

- **Tick vs Update.** Физика и инпут — в `tick`. Анимации и UI — в `update`. Путать чревато.
- **Transform.** Это `{pos = Vec(...), rot = Quat(...)}`, а не просто позиция. `TransformToParentPoint`, `TransformToLocalPoint` — для перевода координат.
- **Регистры — это строки.** `GetFloat`, `SetFloat`, `GetString` работают с ключами вида `"savegame.mod.myvar"`.
- **Производительность:** `QueryRaycast`, `QueryAabbShapes` и подобные — не дёшевы. Не пихать в `tick` без нужды.
- **VOX vs Shape.** VOX — это файл-модель. Shape — это объект в сцене. Не одно и то же.
- **`FindBody("tag", true)` vs `FindBody("tag")`** — второй аргумент означает поиск по всей сцене, иначе только в текущем скрипт-скоупе.
- **Звук:** `LoadSound` грузит, `PlaySound` играет. Грузить — один раз в `init`, играть — где нужно. `LoadSound` в `tick` — классическая ошибка, убивающая FPS.

## 7. Audio Notes (project-specific)

Владелец проекта — звуковой дизайнер AAA-проекта. Это значит:

- Объяснять базу звука в играх не нужно (DSP, бассы/верха, mix, реверб, дакинг — это всё знакомо).
- При работе со звуковыми механиками — предлагай решения уровня саунд-дизайнера, а не «играем wav при коллизии». Динамические лупы, питч-вариации по скорости/массе, импакт-слои, процедурные цепочки.
- Помни ограничения Teardown API по звуку (см. `Docs/audio-notes.md` если файл существует — там накапливаются проектные наблюдения и обходы лимитов API).
